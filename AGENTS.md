# AGENTS.md

## Назначение проекта
Это **одностраничное HTML-приложение** для анализа личных финансовых потоков с русскоязычным интерфейсом.
Главный рабочий файл на текущем этапе: **`v1.0.html`**.

Приложение не является банковским клиентом и не должно развиваться как система учёта баланса счёта. Это аналитический инструмент для доходов, расходов, долговых/исключаемых операций и пользовательских категорий.

## Текущее техническое устройство
- Проект сейчас реализован как **один файл**: HTML + CSS + JS без сборщика и без фреймворков.
- Состояние хранится в `localStorage`.
- Основный ключ хранения: `finance-analyzer-v3`.
- Есть поддержка чтения старого ключа: `finance-analyzer-v2`.
- При изменении структуры данных нужно обновлять миграцию в `ensureDataShape()`.

## Критичные продуктовые правила
Ниже не пожелания, а рабочие ограничения. Если задача конфликтует с ними, приоритет у этих правил.

### 1. Общая продуктовая логика
- Это **анализатор**, а не ledger с балансом счёта.
- Не возвращать логику стартового баланса.
- Не добавлять банковские сущности без прямого запроса.
- Интерфейс и текст — **на русском языке**.
- При отсутствии пользовательских данных в категориях должны оставаться только системные категории, без автогенерации кучи стартовых пользовательских категорий.

### 2. Архитектурные ограничения
- Без прямого запроса **не переводить проект на React/Vue/TypeScript/сборщик**.
- Сохранять рабочую модель **single-file HTML app**, если задача не требует рефакторинга.
- Если рефакторинг всё же нужен, он должен быть оправдан: уменьшение дублирования, лучшая читаемость, безопасная миграция состояния.
- Не ломать обратную совместимость с уже сохранёнными данными.

### 3. Периоды и аналитика
- Период влияет на KPI и аналитику там, где это ожидается по смыслу.
- Формат отображения периода должен быть понятным и человекочитаемым.
- Для полноценных периодов использовать краткие подписи вроде:
  - `февраль 2026`
  - `2 квартал 2026`
  - для диапазона — даты
- Не писать метки периода с лишним визуальным шумом.

### 4. Операции
- Важна удобная быстрая ручная работа с операциями.
- Сортировка, фильтры, пагинация и история последних добавленных операций считаются важными частями UX.
- Для добавления операции нужен toast с понятной фиксацией того, что именно было добавлено.
- По текущему файлу уже используется toast формата:
  - заголовок: `Операция добавлена:`
  - текст: `Категория - Подкатегория - Сумма`
- Не ухудшать скорость ручного ввода.

### 5. Категории и подкатегории
- Системная категория `Подписки` должна существовать всегда.
- Системную категорию нельзя превращать в обычную.
- Для категории расхода допускается флаг `showInComparison`.
- Для категории дохода такой функционал не должен навязываться без смысла.
- У подкатегорий допускается `fixedAmount`, который подставляется в форму операции, но пользователь должен иметь возможность изменить сумму.

### 6. Подписки
`Подписки` — отдельная системная категория расходов.

Требования:
- У подкатегорий подписок могут быть:
  - `Действует до`
  - `Игнорировать просрочку`
- При переносе обычной подкатегории в `Подписки` не нужно навязывать отдельную сложную механику, если можно обойтись простым UX.
- В текущем файле уже используется уведомление:
  - `Перенесено в «Подписки». Чтобы расчёт работал корректно, укажи срок действия в операциях.`
- Подписки должны участвовать в дашборде как отдельный блок, а не теряться среди обычных расходов.

### 7. Сравнение с предыдущим периодом
Блок сравнения — один из ключевых элементов продукта.

Ожидания:
- Заголовок: `Сравнение с предыдущим периодом`
- Не превращать блок в бессмысленную декоративную диаграмму.
- Важны читаемость, сортировка, устойчивость к большим суммам и понятные пустые состояния.
- Не показывать категории, которых не было в предыдущем сопоставимом периоде, если это ломает смысл сравнения.

### 8. Деньги и формат чисел
- Денежные суммы показывать в рублях.
- Не возвращать копейки/дробные хвосты без прямого запроса.
- В текущем файле форматирование уже идёт через `Intl.NumberFormat('ru-RU', { currency: 'RUB', maximumFractionDigits: 0 })`.

### 9. Темы и визуальный стиль
В проекте уже есть темы:
- `Graphite`
- `Light`
- `Cute Cats`
- `Vasik edition`

При правках темы:
- не ломать читаемость светлой темы;
- не терять контраст бейджей и статусов;
- не добавлять визуальный шум ради украшательства.

## Что Codex должен делать при правках
1. Сначала понять, **какая часть поведения уже реализована**, а какая только ожидается.
2. Вносить изменения точечно, без хаотического переписывания половины файла.
3. Если меняется data shape:
   - обновить `defaultData()` при необходимости;
   - обновить `ensureDataShape()`;
   - сохранить совместимость с уже записанными данными.
4. Если меняется UI:
   - проверить связанный рендер;
   - проверить обработчики событий;
   - проверить, не сломались пустые состояния;
   - проверить светлую тему.
5. Если меняется логика подписок или сравнения периодов:
   - проверить данные без операций;
   - проверить данные с одной категорией;
   - проверить данные с несколькими месяцами;
   - проверить перенос подкатегории в `Подписки`.

## Чего делать не нужно
- Не добавлять новые зависимости без явной необходимости.
- Не разводить абстракции ради абстракций.
- Не делать “красивый” рефакторинг, который ломает сохранённые данные.
- Не менять пользовательские формулировки в интерфейсе без причины.
- Не предлагать мобильную версию как обязательную цель: проект ориентирован прежде всего на desktop/laptop UX.

## Как лучше работать с этим проектом
Для любой серьёзной задачи:
1. Сначала коротко перечислить, какие функции и области файла будут затронуты.
2. Потом внести изменения.
3. После изменений явно проверить:
   - загрузку приложения;
   - переключение вкладок;
   - добавление операции;
   - редактирование категории/подкатегории;
   - работу `Подписок`;
   - блок сравнения;
   - сохранение и повторную загрузку через `localStorage`.
